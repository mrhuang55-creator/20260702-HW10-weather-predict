### 7. 非阻塞式背景同步重構
*   **背景**：
    > 第一階段 `check_and_auto_sync()` 在資料過期時需等待 CWA API 回應（3-8 秒），造成前端請求 pending。
*   **實作內容**：
    - 引入 `threading` 模組，將「資料過期時的重新同步」改為背景執行緒（daemon thread），呼叫後立即返回現有快取資料，實現**零延遲回應**。
    - 以 `threading.Lock()` 搭配 `_sync_in_progress` 全域旗標實作互斥鎖，避免多個並發請求重複觸發同步競態。
    - **首次啟動例外**：資料庫為空時仍維持同步阻塞（等待第一筆資料）。
    - 同步修正 `app.py` 第 99 行 JS 風格的 `true` 為 Python 的 `True`。

### 10. 金鑰安全防洩漏與雙層環境設定重構
*   **使用者輸入**：
    > 分析常用文件，並更新所有文檔，避免個資外洩
    > 我如果這樣處理，外部測試當機補救
*   **技術決策與實作**：
    - **安全痛點**：舊 `.env` 檔包含了真實的中央氣象署與 OpenAI API 金鑰，若不慎上傳會造成外洩；但若直接清空金鑰，本機執行或外部自動化測試又會當機。
    - **解決方案**：
      1. 將原本包含真實金鑰的 `.env` 檔在本機重新命名為私有設定檔 `env.private`。
      2. 建立全新的公開範本檔 `.env`，內含金鑰佔位符（`YOUR_CWA_API_KEY_HERE`），可安全提交上傳。
      3. 在 `.gitignore` 中明確加入 `env.private`，徹底防止私有金鑰隨 Git 推送至雲端。
      4. 修改 `app.py`、`weather.py` 及 `ai_helper.py` 頂層的載入邏輯：**優先尋找並讀取 `env.private`，若不存在則 fallback 讀取 `.env`**。
      5. 更新 `README.md` 中的配置指南，詳述本機雙環境檔案配置步驟，確保本機持續以真實 API Key 流暢運行，外部與 Git 端則完全處於無個資安全狀態。
